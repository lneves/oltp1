
# OLTP-1: A TPC-E Inspired Database Benchmark

OLTP-1 is a high-performance, TPC-E-inspired, benchmark tool designed to measure the On-Line Transaction Processing (OLTP) performance of different database systems. Written in Java, it aims to provide a reasonably compliant implementation of the TPC-E workload, focusing on raw transactional performance and cross-database comparability.

This tool is intended for performance analysis and comparison, not for generating officially audited TPC-E results. The workload and results generated by this implementation **do not constitute an official TPC-E result** and are **not comparable** to any published TPC-E benchmarks.

While the older TPC-C benchmark is still more commonly used, TPC-E remains the TPC's official and most realistic standard for evaluating modern OLTP systems and this project implements the TPC-E benchmark with a fresh perspective. The primary motivations were:

* **Enable Cross-Database Comparison**: A core goal is to provide a consistent and fair workload to "shootout" different database technologies. The architecture is explicitly designed with portability in mind, using a dialect-aware query factory to ensure that the same transactional logic can be executed against PostgreSQL, SQL Server, and others, providing a level playing field for performance comparisons.

* **Modernize the Toolchain**: The official TPC-E tools, while compliant, are built in C++ with rigid assumptions about system architectures (e.g., running on Windows, tightly coupled to SQL Server), difficult to extend or modernize without forking and significant engineering work. This project is a chance to leverage something a bit more modern and easier to maintain, extend, and integrate into current development environments.

* **Promote Extensibility**: Core transaction logic is decoupled from the SQL queries, which makes it straightforward to add support for new databases. By simply implementing a new set of dialect-specific queries and registering them in the existing factory system, a new database can be integrated without altering the main benchmark code.

## Features

* **Comprehensive Workload**: Implements all of the TPC-E transactions, including:
    * **Read-Only**: `Broker-Volume` , `Customer-Position` , `Market-Watch` , `Security-Detail` , `Trade-Lookup` , `Trade-Status`.
    * **Read-Write**: `Trade-Order` , `Trade-Result` , `Trade-Update` , `Data-Maintenance` , `Trade-Cleanup`.
* **Configurable**: Workload parameters such as the number of clients and test duration are fully configurable via command-line arguments.
* **Multi-Database Support**: Designed for portability with specific optimizations for:
    * PostgreSQL 
    * Microsoft SQL Server 
    * OrioleDB (via its PostgreSQL compatibility)

* **High-Fidelity Data Generation**: Implements the TPC-E specification's non-uniform random data generation to ensure a realistic and skewed data access pattern.
* **Asynchronous Processing**: Simulates the TPC-E Market Exchange Emulator (MEE) by using an internal thread pool to process some of the transactions asynchronously.
* **Detailed Reporting**: Provides a clear and concise summary report to the console upon completion, detailing TPS, latency percentiles, and error rates for each transaction type.


### Primary Deviations from Specification

The deviations from the official TPC-E specification are generally simplifications or modernizations that make the project more practical.

#### 1. Market Exchange Emulator (MEE) Implementation
* **Deviation**: The TPC-E specification describes the MEE as a separate component that processes trades asynchronously. This implementation simulates this **in-process** using a Java `ExecutorService`. When a `Trade-Order` transaction completes, it submits the `Trade-Result` and `Market-Feed` tasks to this thread pool for deferred execution.
* **Reasoning**: This is a pragmatic simplification. Building a true external MEE application is complex. Using an in-process thread pool achieves the same essential goal—asynchronous processing of trade outcomes—without the overhead of inter-process communication, making the benchmark easier to deploy and run.
One potential issue, however, is the lack of guaranteed ordering. The specification implies a sequence where a trade result might trigger a market feed. By submitting both tasks to a thread pool, their execution order is not guaranteed, which could lead to race conditions or logical inconsistencies if downstream processing depends on this order. For the purpose of this benchmark it's not critical, but it is a deviation from a real-world sequential flow.

#### 2. Transaction Input Generation
* **Deviation**: The official TPC-E EGen tool is a sophisticated state machine that navigates data interdependencies and state transitions to generate transaction inputs. This implementation simplifies this by loading reference data from the database into memory during startup.
* **Reasoning**: Replicating the entire EGen state machine is a massive undertaking. This approach is a significant simplification that still provides the necessary data for generating compliant inputs for the transactions. It captures the essence of the data relationships without the full complexity of the official generator.
At startup the driver populates some internal data strutures by querying the database for essential reference data like customer and company identifiers. To efficiently manage these potentially large datasets without impacting the JVM's garbage collector, this data is stored off-heap.

#### 3. Database Interaction and SQL Dialects
* **Deviation**: For complex inputs, this implementations passes data as **JSON strings** and use database-specific functions to parse them (e.g., `jsonb_to_recordset` in PostgreSQL  and `OPENJSON` in SQL Server ). The TPC-E spec does not define inputs in this manner.
* **Reasoning**: Passing data as JSON simplifies the client-side code by avoiding the need to loop and bind dozens of individual parameters while also avoiding costly client-server roundtrips.

## Getting Started

### Prerequisites

* Java 21 or higher.
* Apache Maven 3.5.x or higher.
* Database client tools (e.g.: psql, sqlcmd, bcp, etc.)
* Docker
* Git

### Setup

I wish this could be easier! :-)

There are thres steps necessary to run the benchmark:

* Generate the data to load in database
* Create the database schema and load the data
* Build and execute the benchmark-driver

Step zero is to clone this repository.

```bash
git clone <repository-url>
cd oltp1
```


#### Data Generation
Currently the "data generation" part is not implemented, we defer to the oficial TPC-E Tools to handle that task. 
The first step is to download the [TPC-E Tools](https://www.tpc.org/tpc_documents_current_versions/current_specifications5.asp) separately and place the file in the `./docker/egen_loader` directory and then to "containerize" the tool.

```bash
cd ./docker/egen_loader
docker build -t egen-loader .
```

After this step you can use the EGenLoader tool to write the data files. 
Below is a list of the most relevant parameters.

--
| Option | Default | Description |
| :--- | :--- | :--- |
| `-b <number>` | 1 | Beginning customer ordinal position |
| `-c <number>` | 5000 | Number of customers (for this instance) |
| `-t <number>` | 5000 | Number of customers (total in the database) |
| `-f <number>` | 500 | Scale factor (customers per 1 tpsE) |
| `-w <number>` | 300 | Number of Workdays (8-hour days) of initial trades to populate |
| `-i <dir>` | `flat_in/` | Directory for input files |
| `-l [FLAT\|ODBC\|CUSTOM\|NULL]` | `FLAT` | Type of load |
| `-m [APPEND\|OVERWRITE]` | `OVERWRITE` | Flat File output mode |
| `-o <dir>` | `flat_out/` | Directory for output files |
---

See a more detailed explanation about the parameters [here](egen.md).

To generate a test dataset with 5000 customers:

```bash
cd ./docker/egen_loader
docker run --rm -v $(pwd)/flat_out:/app/flat_out egen-loader /app/EGenLoader -c 5000 -t 5000
```

Note: According to specfication, the minimum database size must have 5000 customers (see clause 2.6.1.2)

### Database Setup

You can either use the provided `docker-compose.yml` file to quickly spin up a local database instance, or connect to your own pre-configured database server.

 To use the provided setup, follow this steps:

1.  Navigate to the `./docker/postgresql` directory.
2.  Create a `.env` file from the `.env.example` file and customize the values if needed.
3.  Run `docker-compose up`.

### Create the database schema and load the data:

1.  Navigate to the appropriate directory under `./ddl-scripts/` (`postgresql`, `mssql`, or `orioledb`).
2.  Create a `db.properties` file from the `db.properties.example` file and customize the values.
3.  Run the `init_db` script.


### Building the Benchmark-Driver

To build it, run the following Maven command from the project root:

```bash
mvn clean package
```

This will produce an executable shell script in the `./target/` directory.

## How to Run

Execute the benchmark using the `./target/oltp1` command. All parameters are passed via command-line options.

### Command-Line Arguments

| Option              | Description                                                                              | Required | Default |
| ------------------- | ---------------------------------------------------------------------------------------- | :------: | :-----: |
| `-h`, `--host`      | The database server host or IP address.                                       |   **Yes** |   -     |
| `-p`, `--port`      | The database server listening port.                                           |    No    | DB-specific |
| `-U`, `--user`      | The username for the database connection.                                     |   **Yes** |   -     |
| `-P`, `--password`  | The password for the database user.                                           |   **Yes** |   -     |
| `-e`, `--engine`    | The database engine under test. Valid values: `PGSQL`, `MSSQL`.               |   **Yes** |   -     |
| `-d`, `--duration`  | The total duration of the test run in seconds.                                |   **Yes** |  `360`  |
| `-c`, `--clients`   | The number of concurrent simulated clients/users.                             |   **Yes** |  `10`   |
| `-b`, `--baseline`  | If present, runs a simple `SELECT 0;` query instead of the full transaction mix. |    No    | `false` |

### Example Usage

Here is an example of how to run a 10-minute (600 seconds) benchmark with 50 clients against a PostgreSQL database running on localhost.

```bash
./target/oltp1 \
    --host localhost \
    --port 5432 \
    --user tpce \
    --password "db_user_password" \
    --engine PGSQL \
    --clients 50 \
    --duration 600
```
`--engine` can take `PGSQL|MSSQL|ORIOLEDB`

## Benchmark Details

### Transaction Mix

The workload is designed to follow the TPC-E specification's transaction mix closely:

| Transaction         | Mix Percentage |
| ------------------- | :------------: |
| Trade-Status        |     19.0%      |
| Market-Watch        |     18.0%      |
| Security-Detail     |     14.0%      |
| Customer-Position   |     13.0%      |
| Trade-Order         |     10.1%      |
| Trade-Result        |     10.0%      |
| Trade-Lookup        |      8.0%      |
| Broker-Volume       |      4.9%      |
| Trade-Update        |      2.0%      |
| Market-Feed         |      1.0%      |
| Data-Maintenance    |  \~1 per minute |
| Trade-Cleanup       |   Once at start|

*Note: `Trade-Result` and `Market-Feed` are triggered by `Trade-Order` and their percentages are included for completeness.* 

### Reporting

At the end of a run, a summary is printed to the console. The report includes key performance indicators for each transaction type:

* **Target/Actual Mix %**: The configured vs. measured percentage of the transaction in the workload.
* **Response Time (ms)**: Mean, Standard Deviation, Min, Max, and 90th Percentile latency.
* **Count**: Total number of completed transactions.
* **Errors/Warnings/Rollbacks**: A count of transactions that resulted in an error, a warning, or a user-initiated rollback.
* **Rate (tx/sec)**: The throughput for the individual transaction type.

A final summary provides the aggregate transaction rate (TPS) for the entire run.

### Baseline Query Benchmark

The benchmark-driver as an option to run a *Baseline Query* benchmark. This executes a minimal query (e.g., `SELECT 0;`) repeatedly to estimate the theoretical maximum throughput (transactions per second) of the database under ideal, low-latency conditions. This test is useful for establishing an upper bound on performance without the overhead of complex queries, indexing, or data access patterns.

Use this mode to:

* Measure the raw performance ceiling of the database engine and network stack.
* Identify infrastructure bottlenecks unrelated to query complexity.
* Compare different environments or configurations under identical, minimal workloads.
